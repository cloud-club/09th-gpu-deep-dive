## 1. NUMA와 UPI

### NUMA란

- 멀티 소켓 서버에서 메모리 접근 시간이 균일하지 않은 구조 (Non-Uniform Memory Access)
- 각 CPU가 자기 메모리 컨트롤러와 전용 메모리를 직접 보유
- 로컬 접근 = 빠름 / 원격 접근 = 느림

### UPI

- 두 CPU 소켓을 연결하는 고속 인터커넥트 (Intel)
- 반대편 메모리에 가려면 반드시 UPI를 건너야 함 → 대역폭↓, 지연↑
- 로컬: 메모리 전용 채널로 직접 연결 → 넓음
- 원격: UPI를 거침 → 좁음
- UPI가 좁은 이유: 범용 연결선이라 캐시 일관성 신호 등 다른 트래픽도 함께 처리, 물리적 자원도 제한적

### 원격 접근이 잦으면

데이터 이동 지연 → 연산 유닛이 대기 → 처리량 저하

→ 해결: **NUMA-aware 배치** (프로세스와 메모리를 같은 노드에 묶기)
→ 이 원칙은 GPU·NIC 등 PCIe 장치 배치 전략으로도 확장됨

> NUMA는 메모리 접근이 비균일한 구조이며, UPI를 건너는 원격 접근은 대역폭이 줄고 지연이 늘어 성능을 떨어뜨린다.

## 2. NVLink vs PCIe

### PCIe

PCIe는 본래 **CPU ↔ 장치** 범용 통로. GPU끼리 통신하면

`GPU A → PCIe 스위치 / CPU → PCIe 스위치 → GPU B`

1. 우회 경로 → 지연↑
2. SSD·NIC와 통로 공유 → 경합
3. 대역폭 자체가 낮음

### NVLink (GPU 전용 직결 통로)

- CPU/스위치를 거치지 않고 GPU끼리 직접 연결
- HBM에서 곧바로 다른 GPU로 전송
- GPU 간 메모리 일관성 유지

### 

### NVSwitch — 다수 GPU 연결

- GPU 2~4개: NVLink 직접 연결 가능
- GPU 8개+: 모든 쌍 직접 연결 시 링크 수 폭발 (8개 완전 연결 = 28개 연결 필요)
- NVSwitch: 모든 GPU를 중앙 스위치에 연결 → All-to-All 풀 대역폭 통신

> PCIe는 CPU↔장치 범용 통로라 GPU 간 통신에 우회·공유·저대역폭 문제가 있고, NVLink는 GPU 전용 직결로 약 7배 대역폭을 제공하며, NVSwitch가 이를 다수 GPU로 확장한다.

## 3. UPI vs NVLink


|  | UPI | NVLink |
| --- | --- | --- |
| 연결 대상 | CPU ↔ CPU | GPU ↔ GPU |
| 제조사 | Intel | NVIDIA |
| 주 목적 | 캐시 일관성 유지 | 대량 데이터 전송 |

### 왜 둘 다 필요한가 — 트래픽 성격이 다르다

- **CPU 간**: 데이터양 적음 (제어·분기가 본업) → UPI는 정확하고 일관된 소량 통신에 최적화
- **GPU 간**: 거대한 텐서 통째로 전송 → NVLink는 무조건 넓게에 최적화

<br>

### 대역폭 격차

| 항목 | UPI (Sapphire Rapids) | NVLink 4 (H100) |
| --- | --- | --- |
| 연결 대상 | CPU ↔ CPU | GPU ↔ GPU |
| 링크당 대역폭 | ~20+ GB/s | 50 GB/s |
| 링크 수 | 프로세서당 최대 4개 | GPU당 최대 18개 |
| 총 대역폭 | ~100 GB/s 수준 | 900 GB/s |
- 총 대역폭 기준 NVLink가 UPI의 **약 9배**
- NVLink는 Blackwell 1.8 TB/s로 계속 폭증, UPI는 보수적 증가

<br>

### NUMA 관점 연결 — NVSwitch가 UPI 병목을 우회

`nvidia-smi topo -m`의 `SYS` 라벨 = UPI를 건너야 하는 경로

GPU#0(Node0) ↔ GPU#4(Node1) 통신 시, NVLink가 없다면:

`GPU0 → PCIe → UPI(~100 GB/s) → PCIe → GPU4`

- GPU는 900 GB/s로 쏠 준비가 됐는데 중간 UPI가 100 GB/s → **9배 좁은 구간에서 병목**
- NVSwitch: GPU 통신을 NVLink Fabric 안에서 처리 → 노드 경계와 무관하게 풀 대역폭
- `topo -m`에서 GPU0~7이 전부 `NV18`로 직결된 출력 = UPI 병목을 우회한 상태

<br>


> UPI는 CPU 간 캐시 일관성에 최적화된 좁고 정확한 연결, NVLink는 GPU 간 대량 전송에 최적화된 넓은 연결(약 9배)이며, NVSwitch는 GPU 통신이 느린 UPI를 거치지 않게 우회시켜 NUMA 노드 경계의 병목을 제거한다.
>

## 전체 정리

| 연결 | 대상 | 목적 | 대역폭 |
| --- | --- | --- | --- |
| 메모리 채널 | CPU ↔ 자기 메모리 | 로컬 메모리 접근 | 넓음 |
| UPI | CPU ↔ CPU | 캐시 일관성 + 소량 교환 | ~100 GB/s |
| PCIe | CPU ↔ 장치 | 범용 연결 | ~128 GB/s (5.0) |
| NVLink | GPU ↔ GPU | 대량 데이터 직결 | 900 GB/s (Gen4) |
| NVSwitch | 다수 GPU | All-to-All 풀 대역폭 | UPI 병목 우회 |

핵심: 통로마다 **연결 대상**과 **설계 목적**이 다르며, 대량 데이터를 다루는 AI 워크로드일수록 GPU 전용 고대역폭 통로(NVLink/NVSwitch)의 유무가 성능을 좌우한다.

AI 모델을 빠르게 훈련하려면 GPU 간 통신이 중요한데, 이를 제대로 이해하려면 서버 아키텍처부터 알아야 합니다. 오늘 배운 내용이 여러분의 GPU 서버 튜닝에 큰 도움이 될 거라고 생각합니다.

### **Section 1. NUMA와 UPI**

자, 먼저 멀티 소켓 서버를 생각해보세요. 최신 고성능 서버는 보통 2개 이상의 CPU 소켓을 갖고 있습니다.

**NUMA**는 "Non-Uniform Memory Access"의 약자인데, 핵심은 이겁니다:
- 각 CPU 소켓이 자기 전용 메모리 컨트롤러를 가지고 있다
- CPU가 자기 메모리에 접근할 때는 **빠르지만**
- 반대편 CPU 메모리에 접근할 때는 **느리다**

왜냐하면 두 CPU를 연결하는 **UPI**(Ultra Path Interconnect)라는 고속 링크를 거쳐야 하는데, 이게 메모리 채널보다 훨씬 좁거든요.

예를 들어, 로컬 메모리는 CPU가 직접 깔깔거리면서 접근할 수 있는 넓은 도로라면, UPI는 서로 다른 노드를 연결하는 진짜 좁은 다리입니다.

이렇게 되면 뭐가 문제냐? **프로세스가 자꾸 원격 메모리에 접근하면**, 지연이 쌓이고 CPU가 대기 상태에 빠져서 처리량이 확 떨어집니다.

해결책은 **NUMA-aware 배치**입니다. 즉, 프로세스와 메모리를 같은 노드에 묶어두세요. GPU나 네트워크 카드도 마찬가지 원칙입니다.

---

### **Section 2. NVLink vs PCIe**

이제 GPU들이 서로 통신하는 방식을 봅시다.

**PCIe**는 원래 CPU와 SSD, 네트워크 카드 같은 주변 장치를 연결하는 일반 목적 통로입니다. 따라서 GPU가 이 길을 통해 통신하면:

1. GPU A → PCIe 스위치 → CPU → PCIe 스위치 → GPU B 
2. 이 경로는 우회가 많아서 **지연이 늘어나고**
3. SSD나 네트워크 카드와 **대역폭을 공유**하며
4. PCIe 대역폭 자체가 **원래 낮습니다**

**NVLink**는 완전히 다릅니다. 이건 NVIDIA가 GPU끼리 직접 연결하려고 만든 전용 고속 통로입니다.

- GPU에서 바로 다른 GPU로 데이터 날려보낼 수 있고
- CPU를 거칠 필요가 없으며
- GPU 간 메모리 일관성도 자동으로 유지됩니다

근데 여기서 문제가 하나 생깁니다. GPU를 많이 쓸 때예요.

**GPU 2~4개 정도면** NVLink로 직접 연결할 수 있습니다. 하지만 **GPU 8개 이상**이 필요한 대규모 학습에서는, 모든 쌍을 직접 연결하려면 필요한 링크가 폭발적으로 증가합니다.

이때 등장하는 게 **NVSwitch**입니다. 중앙 스위치에 모든 GPU를 연결해서 All-to-All 풀 대역폭 통신을 가능하게 합니다. 마치 데이터센터 스위치처럼요.

정리하면:
- PCIe: 우회, 공유, 저대역폭 → 낮은 성능
- NVLink: 직결, 전용, 고대역폭 → 높은 성능
- NVSwitch: 직결의 확장 → 다수 GPU도 대역폭 보장

---

### **Section 3. UPI vs NVLink — 왜 둘 다 필요한가?**

여기서 중요한 질문이 생깁니다. **"그럼 NVLink가 그렇게 좋으면, CPU 간 통신도 NVLink처럼 하면 안 되나?"**

답은 **트래픽 성격이 다르기 때문**입니다.

**CPU 간 통신**을 보세요. 대부분이 무엇인가요? 캐시 일관성 신호, 제어 신호, 메모리 잠금 같은 **작고 정확한 통신**입니다. 데이터양은 적지만 **신뢰성과 순서**가 중요합니다.

**GPU 간 통신**은 뭔가요? 거대한 행렬 텐서, 활성화 맵, 그래디언트—**메가바이트 단위의 덩어리**를 곧바로 쏟아붓습니다.

따라서:
- UPI는 정확하고 일관된 **소량 통신에 최적화**
- NVLink는 무조건 **넓게만 최적화**

**대역폭을 보면 격차가 더 확연합니다.**

UPI는 링크당 약 20 GB/s 정도인데, GPU마다 최대 4개 정도? NVLink 4는 링크당 50 GB/s인데 GPU마다 **최대 18개**입니다.

계산해보면:
- UPI 총합: ~100 GB/s 수준
- NVLink 총합: 900 GB/s

**약 9배 격차**입니다. Blackwell이 나오면 NVLink는 1.8 TB/s까지 뛰는데, UPI는 보수적으로 증가합니다.

---

### **Section 4. NVSwitch가 NUMA 병목을 우회하는 방식**

이제 더 깊이 들어가봅시다.

2-소켓 서버에 8개의 GPU가 있다고 생각해보세요. GPU 0~3은 Node 0에, GPU 4~7은 Node 1에 물려 있습니다.

**`nvidia-smi topo -m`을 실행하면**, SYS 라벨이 붙은 경로가 보입니다. 이게 뭔가요? **UPI를 건너야 하는 경로**입니다.

만약 NVSwitch가 **없다면**, GPU 0과 GPU 4 간 통신은:
- GPU 0에서 테이터 뽑아내기
- PCIe 거쳐서
- UPI로 노드 경계 넘어가기 (~100 GB/s)
- 다시 PCIe
- GPU 4에 도착

GPU 자체는 900 GB/s로 쏟을 준비가 되어 있는데, **중간에 100 GB/s의 좁은 UPI 병목**이 생깁니다.

**NVSwitch가 있으면** 상황이 다릅니다. GPU 간 모든 통신이 NVLink 패브릭 **안에서만** 일어나므로, 노드 경계를 무시하고 풀 대역폭을 유지합니다.

이걸 확인하는 방법은? `topo -m` 출력에서 GPU 0~7이 전부 **`NV18`** 같은 라벨로 직결되어 있으면, UPI 병목을 우회한 상태입니다.

---

### **마무리**

정리하면:

**NUMA**: 메모리 접근이 비균일하니, 프로세스와 메모리를 같은 노드에 배치하세요.

**UPI**: CPU 간 캐시 일관성에 최적화된 좁은 연결. 작고 정확한 통신용.

**NVLink + NVSwitch**: GPU 간 대량 전송에 최적화된 **약 9배 넓은 연결**. AI 워크로드 성능의 핵심.

결론: **통로마다 설계 목적이 다릅니다**. 멀티 GPU 시스템을 튜닝할 때는 이런 아키텍처를 이해하고, 데이터가 최적의 경로를 가도록 배치해야 최고 성능을 얻을 수 있습니다.
