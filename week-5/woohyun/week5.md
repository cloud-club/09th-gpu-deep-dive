## 1. LLM이 GPU를 쓰는 두 가지 상황

LLM이 GPU를 사용하는 상황은 크게 두 가지다.

- **학습** (시험 공부)
- **추론** (시험 보기)

### 학습 : 모델이 공부를 하는 단계

- 문제를 많이(큰 batch) 푼다
- 정답이랑 비교해서 "얼마나 틀렸나"(loss)를 잰다
- 틀린 만큼 자기 머릿속 지식(weight)을 고친다
- 이걸 계속 반복

weight를 계속 바꾸기에 어디를 얼마나 고칠지(gradient), 고치는 작업용 메모(optimizer state)를 들고 있어야 하여 메모리에 weight, gradient, optimizer를 올린다.

### 추론 : 모델이 시험 보는 단계

공부를 끝낸 모델한테 질문을 하면 답을 얻는 것. 지식(weight)은 고정되어 있어 바꾸지 않고 해당 지식으로 답(token)만 뽑아낸다. 답을 뽑아내면서 쌓이는 메모인 **KV cache**를 가진다.

- 사용자가 질문하면 첫 글자를 빨리 내놓는다.
- 위 과정 이후부터는 일정한 간격으로 다음 글자들을 계속 내놓는다.

### 병목(느려지는 지점)이 다르다

- **학습의 병목** = GPU 간의 동기화 (한 명 늦으면 다 같이 기다림)
- **추론의 병목** = 사용자가 느끼는 지연 (줄 서는 시간, 메모 크기, 글자 뽑는 리듬)

---

## 2. 추론을 한 문장으로 보기

> 추론은 질문(prompt)을 잘게 쪼개고(token), 모델을 반복 통과시켜 다음 글자 확률을 계산한 뒤, 하나씩 골라 뽑는 과정이다.

```
사용자 질문
  → tokenize        (문장을 token 단위로 쪼갬)
  → prefill         (질문 전체를 한꺼번에 읽고, 메모장 KV cache 생성)
  → decode loop     (메모장 보면서 다음 글자를 1개씩 생성)
  → sampling        (후보 중 실제 다음 글자 선택)
  → 답을 화면에 흘려보냄
```

여기서 제일 중요한 건 **prefill과 decode가 같은 모델을 지나지만 느려지는 이유가 서로 다르다**는 것.

## 3. Prefill vs Decode 

비유로 보면 이해가 빠르다.

### Prefill = 시험지 100문제를 한 번에 푸는 것

질문이 8,000 token이면, 그 8,000개를 동시에 모델에 통과시켜 메모장(KV)을 만든다.

- 계산할 게 산더미 → GPU의 "계산기"(Tensor Core)를 꽉 채워 돌림
- 즉 **계산이 병목**
- 나빠지는 지표: **TTFT** (첫 글자 나오는 속도)

### Decode = 한 문제 풀 때마다 두꺼운 참고서를 처음부터 다시 펴는 것

답은 한 글자씩 나온다. 두 번째 글자는 첫 번째를 보고, 세 번째는 1·2번째를 보고, 이렇게 앞 글자에 의존해서 한꺼번에 못 만든다.

```
step 1: 글자1 생성
step 2: 글자1 보고 → 글자2 생성
step 3: 글자1,2 보고 → 글자3 생성
...
```

- 한 글자 계산 자체는 별거 아닌데, 매번 모델 전체(weight)랑 그동안 쌓인 메모(KV)를 읽어와야 함
- 계산기는 한가한데 "읽어오는 속도"가 발목 잡음 → **메모리 읽기 속도(HBM bandwidth)가 병목**
- 나빠지는 지표: **TPOT / ITL** (글자와 글자 사이 간격)

### 정리

| | Prefill | Decode |
|---|---|---|
| 비유 | 100문제 한 번에 풀기 | 한 문제마다 참고서 다시 펴기 |
| 하는 일 | 질문 전체를 한꺼번에 처리 | 답을 글자 1개씩 생성 |
| 병목 | 계산 (Tensor Core) | 메모리 읽기 (HBM bandwidth) |
| 지표 | TTFT (첫 글자) | TPOT/ITL (글자 간격) |

> **기억할 것:** prefill 느리면 계산 문제, decode 느리면 메모리 문제. 튜닝 방법이 다르다.

## 4. 모델 한 층(layer) 안에서 일어나는 일

모델은 똑같은 층을 수십 번 반복 통과한다. 한 층 안에서 하는 일들:

| 연산 | 쉬운 설명 | 병목 |
|---|---|---|
| Embedding | token(숫자)을 의미 벡터로 바꿈 (거대한 표에서 한 줄 찾기) | 메모리 읽기 |
| Norm | 값이 너무 커지거나 작아지지 않게 안정화 | 계산보다 읽고쓰기 |
| Q/K/V 생성 | attention에 쓸 재료 3개 만들기 | prefill은 큰 계산, decode는 작은 계산 |
| Attention | 지금 글자가 과거 글자 중 뭘 볼지 결정 | decode에선 과거 메모(KV) 읽느라 메모리 대역폭 씀 |
| MLP/FFN | 글자별로 정보를 다시 가공 | prefill은 계산, decode는 weight 읽기 |
| Logits/Sampling | 다음 글자 후보 점수 매기고 고르기 | 마지막 변환 + 고르는 비용 |

### Attention을 한 줄로

- **Q** = 지금 글자가 던지는 질문
- **K** = 과거 글자들의 색인(목차)
- **V** = 과거 글자들의 실제 내용

Q와 K를 비교해 "어떤 과거 글자를 많이 볼지" 정하고, 그 비중대로 V를 모아서 다음 상태를 만든다.

### 다음 글자를 고르는 마지막 단계 

```
hidden 상태
  → logits   : 각 단어 후보의 raw 점수 (아직 확률 아님)
  → top-k / top-p / temperature 같은 후처리
  → softmax  : 점수를 확률로 변환
  → sampling : 확률에서 실제 다음 글자 선택 (greedy=제일 높은 거, top-k/p=후보 제한 후 뽑기)
```

> `logits`, `softmax`, `sampling`은 서로 다른 단계다. (같은 말 아님)

## 5. KV Cache : 편한데 무거운 메모장

decode할 때 매번 질문 전체를 다시 계산하면 너무 비싸다. 그래서 한 번 계산한 과거 글자의 정보(Key/Value)를 메모해두는 게 **KV cache**다.

- **좋은 점:** 계산 다시 안 해도 됨
- **나쁜 점:** 이 메모를 GPU 메모리에 계속 들고 있어야 함. 그리고 무섭게 커진다.

### 얼마나 커지나 (선형으로 증가)

```
KV 크기 = 층 수 × batch × 글자수(seq_len) × 2(K,V) × KV head 수 × head 차원 × 자료형 크기
```

기억할 포인트:

- 질문 길이 4배 → KV 4배
- 동시 요청 2배 → KV 2배
- KV head 수 줄이면 → KV 줄어듦

### KV 줄이는 모델 구조: MHA → GQA → MQA

| 방식 | 의미 | KV 크기 |
|---|---|---|
| MHA | head마다 KV 따로 | 큼 |
| GQA | 여러 head가 KV 공유 | 줄어듦 |
| MQA | KV 거의 1개로 공유 | 많이 줄어듦 |

GQA/MQA는 decode 병목 완화에 중요하다 (저장량과 읽는 양이 동시에 감소).

### 크기 감각 (BF16, 요청 1개 기준)

| 모델 | 8k | 32k | 128k |
|---|---|---|---|
| 8B | ~1 GiB | ~4 GiB | ~16 GiB |
| 70B | ~2.5 GiB | ~10 GiB | ~40 GiB |
| 405B | ~3.9 GiB | ~15.8 GiB | ~63 GiB |

> **운영 핵심:** "모델이 GPU에 들어가나?"만 보면 부족하다. 실제론 weight + KV cache + 작업공간 + 여유공간이 다 GPU 메모리에 들어가야 한다. 안 들어가면 OOM(메모리 터짐).

### KV 줄이는 트릭

| 기법 | 뭘 해결 |
|---|---|
| PagedAttention | 메모리를 알뜰히 쪼개 써서 낭비 방지 |
| Prefix caching | 똑같은 앞부분(시스템 프롬프트 등)은 한 번만 계산, 재활용 → TTFT 개선 |
| KV quantization | 메모를 저용량 형식으로 → 저장+읽기 줄임 |
| CPU/NVMe offload | 안 쓰는 KV만 멀리 치움 (자주 쓰는 건 위험, 느려짐) |

## 6. 속도를 재는 시계들

하나의 숫자로 "빠르다/느리다" 못 따진다.

| 지표 | 의미 | 좌우하는 단계 |
|---|---|---|
| TTFT | 첫 글자까지 걸리는 시간 | 줄서기 + tokenize + prefill |
| TPOT/ITL | 글자와 글자 사이 간격 | decode loop |
| Latency | 전체 완료 시간 = TTFT + (글자수 × TPOT) | 둘 다 |
| Throughput | 초당 처리량 (요청/토큰) | batch, GPU 활용도 |
| p99 | 운 나쁜 요청(100명 중 99등)의 느림 = 체감 | 줄서기, 긴 질문 |

### 중요한 함정 (trade-off)

```
batch(한 번에 처리하는 묶음)를 키우면
  → GPU 활용도↑, 전체 처리량↑
  → 하지만 줄서는 시간↑, KV 메모↑
  → 첫 글자(TTFT)와 체감(p99)은 오히려 나빠질 수 있음
```

처리량과 응답속도는 줄다리기 관계. 서비스 종류마다 우선순위가 다르다:

| 서비스 | 먼저 챙길 것 |
|---|---|
| 챗봇/코딩 도우미 | TTFT, 글자 간격, p99 |
| RAG/문서 QA | TTFT |
| 에이전트/툴 사용 | 전체 p95, 글자 간격 |
| 오프라인 대량처리 | 처리량, 비용 (응답속도 양보) |

## 7. Batching : 요청을 어떻게 묶나

### Static batching (옛날 방식)

한 묶음이 다 끝날 때까지 기다림.

```
Req A: [작업 작업 작업 작업]
Req B: [작업 작업 빈칸 빈칸]   ← 먼저 끝나도 못 빠져나감
Req C: [작업 대기 대기 대기]   ← A 끝날 때까지 대기
```

문제: 짧은 요청이 긴 요청을 기다림, 빈칸/대기로 GPU 낭비.

### Continuous batching (요즘 방식)

decode 한 바퀴 돌 때마다 끝난 요청 빼고 새 요청 넣음.

```
t1: A decode | B decode | C prefill
t2: A decode | C decode | D prefill   ← B 끝나자마자 D 투입
```

GPU를 쉬지 않게 채움. (단, 무작정 batch 키우면 KV/줄서기 늘어남)

### Chunked prefill

긴 질문 하나가 prefill을 독점하면 다른 사람 답 쓰기(decode)가 멈춘다. 그래서 긴 질문을 토막 내서 decode 사이사이에 끼워 넣는다.

```
긴 질문 P = P1 + P2 + P3
→ P1 prefill | A decode | B decode | P2 prefill | ...
```

첫 글자(TTFT)와 글자 간격(TPOT) 사이의 균형을 맞춤.

## 8. GPU 한 대 안에서의 병목 (Single-node)

GPU 안에는 여러 길이 있고, 그 길마다 속도가 다르다.

```
GPU HBM (제일 빠름, 가까움)
  ↕ NVLink   (GPU끼리 연결)
  ↕ PCIe     (CPU랑 연결)
  ↕ CPU DRAM (큼, 느림)
  ↕ NVMe     (제일 큼, 제일 느림)
```

### A100 vs H100 감각 (둘 다 80GB지만 다름)

| 항목 | A100 | H100 | 영향 |
|---|---|---|---|
| 메모리 대역폭 | ~2.0 TB/s | ~3.35 TB/s | decode 속도 직결 |
| 계산력 | 낮음 | 높음 | prefill 속도 |
| NVLink | ~600 GB/s | ~900 GB/s | GPU 간 통신 |

### 빠른 진단

- Prefill 느림 → 계산력 / prompt 길이
- Decode 느림 → 메모리 대역폭 / KV / batch
- GPU 간 통신 느림 → NVLink/통신 경로
- offload 느림 → PCIe / DRAM (멀어서 느림)

> offload(멀리 치우기)는 자주 쓰는 데이터엔 쓰면 안 된다. 안 쓰는 데이터(cold) 치우는 보조 수단으로만.

## 9. 여러 GPU로 나누는 3가지 방법 (Multi-node)

모델이 너무 커서 GPU 한 장에 안 들어가면 쪼갠다. 어느 방향으로 자르냐가 핵심.

### TP (Tensor Parallelism) — 옆으로 자르기

한 층(layer) 안의 계산을 여러 GPU가 나눠 함.

```
Layer 1: GPU0 | GPU1 | GPU2  ← 한 층을 셋이 나눠 계산
```

- 층마다 결과를 합쳐야 함 → GPU 간 통신이 매우 잦음
- 규칙: 반드시 NVLink로 빠르게 연결된 GPU끼리 묶어라 (노드 밖으로 새면 느려짐)

### PP (Pipeline Parallelism) — 위아래로 자르기

층을 stage로 나눠 GPU마다 다른 층을 맡음.

```
GPU0: Layer 1~10
GPU1: Layer 11~20
GPU2: Layer 21~30
```

- activation을 다음 stage로 넘김
- 문제: 일감 분배가 안 맞으면 GPU가 논다(pipeline bubble)
- 모델이 한 노드에 아예 안 들어갈 때 사용

### DP (Data Parallelism) — 통째로 복사

모델 전체를 여러 GPU에 복제하고 요청만 나눠 보냄.

```
복제0: 모델 전체 → 요청 A
복제1: 모델 전체 → 요청 B
```

- 제일 단순. 모델이 한 장에 들어가는데 사용자만 많을 때 강력
- 단점: 복제마다 weight+KV를 따로 들어야 함

### 언제 뭘 쓰나

| 상황 | 선택 |
|---|---|
| 7B/8B (한 장에 들어감) | single GPU 또는 DP |
| 70B (한 장에 안 들어감) | 노드 안에서 TP=2/4 |
| 405B+ (노드 하나로도 빠듯) | TP+PP, 양자화 |
| 긴 context로 KV가 큼 | GQA/MQA, KV quant, offload |

## 10. PD 분리 : 일은 잘하는데 새 골칫거리

같이 돌릴 때(colocated): prefill(계산 많음)과 decode(메모리 많고 예민함)를 같은 GPU에서 섞으면, 긴 질문 처리(prefill)가 답 쓰기(decode)를 막아서 답이 뚝뚝 끊긴다.

그래서 **분리(PD disaggregation)**: prefill 전담 GPU / decode 전담 GPU로 나눈다.

- **좋아진 점:** 서로 안 방해함, 각각 다르게 최적화 가능
- **새 문제:** prefill이 만든 KV 메모를 decode GPU로 네트워크로 옮겨야 함 → 이게 새 병목

### KV 옮기는 데 걸리는 시간 (70B 기준)

| Context | KV 크기 | 100G망 | 400G망 |
|---|---|---|---|
| 8k | ~2.5 GiB | ~268 ms | ~67 ms |
| 32k | ~10 GiB | ~1.07 s | ~268 ms |
| 128k | ~40 GiB | ~4.3 s | ~1.07 s |

> 32k만 넘어도 옮기는 시간이 첫 글자 속도(TTFT)를 잡아먹을 수 있다. 128k면 빠른 망이라도 1초급.
