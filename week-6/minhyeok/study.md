# Google TPU / XLA 아키텍처 스터디

> NVIDIA GPU 추론에 익숙한 사람이, TPU 스택을 "작은 예제를 손으로 굴리는" 방식으로 바닥부터 이해하기 위한 자료.
> 5개 세션: 시스톨릭 어레이 → XLA 컴파일러 → 분산 병렬화/GSPMD → ICI/OCS 네트워크 → Ironwood 추론 경제학.

---

## 0. 관통하는 대전제

모든 세션이 사실 **하나의 명제**의 변주다. 시작 전에 이것만 머리에 박는다.

> **칩에서 진짜 비싼 건 계산이 아니라 "메모리(HBM)에서 숫자를 꺼내오는 것"이다.**

곱셈 한 번은 거의 공짜인데, HBM에서 값 하나를 읽는 건 그 수백 배 비싸다 — 전력도 시간도. 그래서 가속기 설계의 게임은 **"같은 숫자를 메모리에서 몇 번 읽느냐"를 줄이는 것**이 전부다.

TPU 스택 전체를 꿰는 두 문장:
1. **메모리 왕복을 없애라.**
2. **하드웨어는 경직되게, 지능은 컴파일러로 올려라.**

NVIDIA는 정반대 베팅이다 — 유연한 하드웨어 + 사람이 손으로 짠 커널(cuDNN/CUTLASS) + 거대 생태계.

## 1. 시스톨릭 어레이 (Systolic Array)

### 1.1 왜 존재하나 — GPU와의 대비

**GPU 방식.** 코어 수천 개에 곱셈-덧셈(MAC)을 잘게 뿌린다. 각 코어가 메모리에서 값을 가져와 곱하고 더하고 다시 쓴다. 문제는 **같은 가중치가 여러 코어에 계속 다시 필요**하다는 것. HBM이 머니까 이를 가리려고 거대한 장치를 단다 — 캐시(L2/셰어드메모리), 워프 스케줄러, 큰 레지스터 파일. 유연한 대신, 칩 면적을 이 "관리 장치"에 많이 쓴다.

**시스톨릭 방식.** 작은 곱셈기들을 격자(grid)로 깔고 **옆 칸끼리 직접 배선**한다. 가중치를 칸에 한 번 박고, 입력을 흘려보내서, 중간값이 메모리를 거치지 않고 칸에서 칸으로 직접 넘어가게 한다. 칸은 멍청한 대신, 면적을 거의 전부 **곱셈기**로 쓴다.

### 1.2 "격자"의 실체

격자 = **곱셈기(MAC 유닛)를 2D로 놓고 옆칸끼리 전선으로 이은 하드웨어.** 각 칸이 하는 일은 평생 딱 한 줄:

```
왼쪽에서 온 입력 a × 내가 박고 있는 가중치 w + 위에서 내려온 부분합 p
= 새 부분합 p′   →  p′는 아래로, a는 오른쪽으로 넘김
```

**weight-stationary**: 가중치를 칸에 박아 고정하고(stationary), 입력을 흘린다. 한 번 적재한 가중치는 뒤따라 흘러드는 모든 입력에 재사용된다 → **다시 안 읽는다.**

### 1.3 3×3 예제 — 클럭 단위로

계산:
```
A = [1 2 3]      B = [2 1 1]
    [...]            [1 2 1]
                     [1 1 2]
```

**Step 1. B를 격자에 박는다** (칸(행k,열j) = B[k,j], 고정):
```
        열1   열2   열3
행1:    [2]   [1]   [1]
행2:    [1]   [2]   [1]
행3:    [1]   [1]   [2]
```

**Step 2. A의 첫 행 [1,2,3]을 왼쪽에서 흘린다.** 값마다 들어가는 줄이 다르고, **한 클럭씩 늦게(staggering)** 투입:
- `1` → 행1에 t1 투입
- `2` → 행2에 t2 투입
- `3` → 행3에 t3 투입

**클럭별 동작표** (C의 첫 행 생성):

| 클럭 | 칸 | a (왼쪽) | p (위에서) | 계산 p + a×w | p′ 행선지 |
|---|---|---|---|---|---|
| t1 | 행1열1 | 1 | 0 | 0 + 1×2 = **2** | ↓ 행2열1 |
| t2 | 행1열2 | 1 | 0 | 0 + 1×1 = **1** | ↓ 행2열2 |
| t2 | 행2열1 | 2 | 2 | 2 + 2×1 = **4** | ↓ 행3열1 |
| t3 | 행1열3 | 1 | 0 | 0 + 1×1 = **1** | ↓ 행2열3 |
| t3 | 행2열2 | 2 | 1 | 1 + 2×2 = **5** | ↓ 행3열2 |
| t3 | 행3열1 | 3 | 4 | 4 + 3×1 = **7** | ★ 출력 C[1,1]=7 |
| t4 | 행2열3 | 2 | 1 | 1 + 2×1 = **3** | ↓ 행3열3 |
| t4 | 행3열2 | 3 | 5 | 5 + 3×1 = **8** | ★ 출력 C[1,2]=8 |
| t5 | 행3열3 | 3 | 3 | 3 + 3×2 = **9** | ★ 출력 C[1,3]=9 |

활성 칸 수: **1 → 2 → 3 → 2 → 1** (데이터가 대각선 파도처럼 격자를 쓸고 지나간다 = 채워지고 비워짐).

**검산** (일반 행렬곱):
```
C[1,1] = 1×2 + 2×1 + 3×1 = 7  ✓
C[1,2] = 1×1 + 2×2 + 3×1 = 8  ✓
C[1,3] = 1×1 + 2×1 + 3×2 = 9  ✓
```

### 1.4 스태거링이 왜 필요한가

t3의 **행3열1**에서 `4 + 3×1 = 7`이 되려면 두 값이 **같은 클럭에 같은 칸에서** 만나야 한다:
- 위에서 내려온 부분합 `4` (행2열1이 t2에 만들어 내려보냄)
- 왼쪽에서 온 입력 `3` (행3에 t3 투입)

`3`을 `1`과 같은 t1에 넣었다면, `3`은 이미 t1에 그 칸을 지나가 버려서 t3에 도착하는 부분합 `4`와 못 만난다. **각 행을 한 박자씩 늦게 투입해야 내려오는 부분합과 들어가는 입력의 타이밍이 맞는다.**

### 1.5 제어는 어떻게 하나 — 핵심 반전

**각 칸을 개별 제어하지 않는다. 못 한다. 그게 핵심이다.**

칸 내부엔 프로그램 카운터도, 분기(if)도, 명령어 페치도 없다. 매 클럭 똑같은 `p + a×w`만 하는 고정 회로다. 바깥에서 거는 제어는 단 두 개:

1. **전역 클럭** — 모든 칸이 한 박자에 동시에 한 번씩 동작(lockstep).
2. **투입 타이밍(스태거)** — 어느 행에 언제 입력을 들이미느냐.

그럼 제어가 사라졌나? 아니다, **위(컴파일러)로 올라갔다.** 진짜 행렬은 4096×4096인데 어레이는 256×256뿐이다. 큰 행렬을 256 조각으로 자르고, 어떤 블록을 먼저 박고, 어떤 순서로 흘리고, 부분 결과를 어떻게 합칠지 — 이 **순서표 전체를 컴파일러(XLA)가 미리 짠다.** 하드웨어는 멍청하게 실행만 한다.

> GPU는 이런 결정을 런타임에 스케줄러가 즉석에서 한다. TPU는 컴파일 타임에 컴파일러가 통째로 짠다. **"하드웨어가 경직될수록 지능이 컴파일러로 올라간다"**의 첫 번째 실례.

### 1.6 강점 / 약점

**강점**
- 중간값(부분합)이 메모리를 안 거치고 칸에서 칸으로 직접 넘어감 → 메모리 왕복 제거.
- 한 번 박은 가중치를 흘러드는 입력 전체에 재사용 → 메모리 읽기 급감 (256×256이면 가중치 하나가 수백 번 재사용).
- 칸에 제어 장치가 없어 면적을 거의 전부 곱셈기로 사용 → 광고 peak 성능에 실측이 가깝게 붙음 (GPU의 "이론 대비 실측 갭"이 작아짐).

**약점**
- **fill/drain 레이턴시**: N×N 어레이는 채우고 비우는 데 ~2N-1 사이클. 작은 행렬은 파이프라인 비용을 못 갚는다.
- **shape 정합성**: 차원이 N의 배수로 안 떨어지면 패딩 → 0을 곱하는 헛된 MAC.
- 결론: **크고 dense하고 규칙적인 행렬곱일 때만 이득.** 작거나 불규칙하면 어레이가 텅 빈 채 돌아 오히려 손해. (→ 이 약점이 Session 5의 decode 문제로 이어진다.)

## 2. XLA 컴파일러 — 연산 융합

### 2.1 정의

> **XLA = 연산 그래프 전체를 미리 받아, 칩이 멍청하게 실행만 하면 되는 고정된 일정표로 빚어주는 컴파일러.**

"순서표를 짠다"의 가장 중요한 트릭이 **연산 융합(operator fusion)**.

### 2.2 융합 예제

계산 (전부 길이 N 벡터, 원소별):
```
y = relu(a * b + c)
a=[1,2,3], b=[2,2,2], c=[1,1,1]

t1 = a * b   = [2, 4, 6]
t2 = t1 + c  = [3, 5, 7]
y  = relu(t2)= [3, 5, 7]
```

대전제대로 **HBM을 몇 번 읽고 쓰는지**만 센다.

**방식 1: 융합 없이** (PyTorch eager 기본 — 연산을 한 줄씩 따로 실행)

| 연산 | HBM 읽기 | HBM 쓰기 | 트래픽 |
|---|---|---|---|
| t1 = a*b | a, b | t1 | 3N |
| t2 = t1+c | t1, c | t2 | 3N |
| y = relu(t2) | t2 | y | 2N |
| | | **합계** | **8N** |

중간값 t1, t2를 HBM에 **썼다가 바로 다시 읽는다** = 순수 낭비.

**방식 2: 융합** (XLA — 셋이 전부 원소별이고 줄줄이 이어지니 한 커널로 합침)

```
원소마다: 1*2=2 → +1=3 → relu=3 → y에 쓰기  (중간값 2,3은 레지스터에만 잠깐)
```

| 융합 커널 | HBM 읽기 | HBM 쓰기 | 트래픽 |
|---|---|---|---|
| relu(a*b+c) | a, b, c | y | **4N** |

**8N → 4N (절반).** 사라진 4N이 정확히 중간값 t1, t2의 왕복.

### 2.3 시스톨릭과 같은 철학

시스톨릭에서 부분합이 메모리 안 거치고 아래 칸으로 넘어간 것과 **완전히 같은 아이디어**다. 거긴 하드웨어 배선으로, 여긴 컴파일러 소프트웨어로. 둘 다 한 문장: **중간값을 HBM 보내지 말고 칩 안에서 바로 다음 단계로 넘겨라.**

> 참고: **FlashAttention**이 바로 이 융합의 극단적 수작업 버전. 어텐션의 여러 단계를 한 커널로 융합해 거대한 중간 행렬을 HBM에 안 쓰게 만든 것. NVIDIA에선 사람이 손으로, TPU에선 XLA가 자동으로 — 같은 목표, 다른 주체.

### 2.4 왜 TPU에서 사활인가

시스톨릭 어레이는 큰 matmul 괴물이지만, 신경망엔 matmul 사이사이에 relu·add·정규화 같은 **메모리 바운드 잔연산**이 잔뜩 껴 있다. 융합 없이 하나씩 돌리면 그 잔연산이 HBM을 들락거리는 동안 **거대한 어레이가 굶는다.** XLA가 잔연산을 뭉쳐 어레이를 안 굶기는 게 핵심 임무. 이를 잘하려면 연산 하나가 아니라 **그래프 전체**를 봐야 한다 = "whole-program AOT 컴파일".

### 2.5 대가

미리 통으로 짜려면 **모양(shape)이 컴파일 타임에 고정**돼야 한다:
- static shape 강제 (텐서 크기 고정).
- 컴파일이 느리고, 모양 바뀌면 재컴파일.
- 중간값이 메모리에 안 남아 디버깅이 까다로움.

이 "모양 고정" 요구가 추론에서 골칫거리가 된다 (가변 시퀀스 길이) → Session 5에서 재등장.

## 3. 분산 병렬화와 GSPMD

### 3.1 왜 칩 하나에 안 들어가나 — 학습 메모리

추론은 가중치만 있으면 되지만, **학습은 파라미터당 4덩어리**다. 파라미터 수 P, Adam + 혼합정밀 기준:

| 덩어리 | 내용 | 크기 (P=10억) |
|---|---|---|
| 가중치 | 모델 파라미터 | 2 B/개 → 2GB |
| 그래디언트 | 역전파 기울기 | 2 B/개 → 2GB |
| 옵티마이저 상태 | momentum + variance + fp32 사본 | 12 B/개 → 12GB |
| | **합계** | **약 16 B/개 → 16GB** |

10억(작은 축)인데 16GB. 70B면 1TB+. H100 한 장은 80GB → **절대 안 들어간다.** 이게 분산의 출발점.

### 3.2 병렬화 4종 — 무엇을 쪼개느냐

레이어 4개 모델 `L1→L2→L3→L4`, GPU 2장 기준.

| 방식 | 무엇을 쪼개나 | 모델이 메모리에 | 통신 | 계열 |
|---|---|---|---|---|
| **DP/DDP** | 데이터(배치) | 칩마다 전체 복제 | 그래디언트만 (적음) | 데이터 병렬 |
| **FSDP/ZeRO** | 데이터 + 모델 보관 방식 | 칩마다 1/N (필요시 모음) | 레이어마다 모음 (중간) | 데이터 병렬 |
| **TP** | 레이어 **내부** 행렬 (가로) | 칩마다 1/N | 레이어마다 합침 (많음) | 모델 병렬 |
| **PP** | 레이어 **사이** (세로) | 칩마다 구간만 | 구간 경계에서 (적음) | 모델 병렬 |

- **DP**: 모델 통째 복제, 배치만 분할. 메모리 안 줄고 처리량만 ↑.
- **TP**: 레이어 하나를 둘이 분업 → 결과가 반쪽씩 나옴. 통신 잦아 보통 노드 안(NVLink)에서만.
- **PP**: 레이어를 덩어리로 나눠 칩마다 다른 구간 담당 (공장 라인).

### 3.3 FSDP / ZeRO 상세

핵심: **평소엔 1/N로 쪼개 들고, 레이어마다 그것만 잠깐 모았다 버린다.**

```
보관:  GPU0 = 각 레이어의 절반,  GPU1 = 나머지 절반
계산:  레이어1 차례 → 둘이 레이어1 조각 교환(all-gather) → 레이어1 완성(잠깐) → 계산 → 버림
       레이어2 차례 → 레이어2만 모음 → ... → 버림
```

**메모리 계산** (레이어 10개, 각 2GB, 총 20GB, GPU 2장):

| | 항상 떠 있는 모델 | 피크 |
|---|---|---|
| DDP (복제) | 전체 20GB | **20GB** |
| FSDP | 내 절반 10GB + 레이어 1개 2GB | **12GB** |

**FSDP 피크 = (전체 ÷ 칩수) + (레이어 1개치).** 전체 20GB가 동시에 올라온 적이 없다. 칩 8장이면 영구 2.5GB + 임시 2GB = **4.5GB** (칩 늘수록 영구분이 0으로 수렴, 남는 건 "레이어 한 개 임시분").

> **FSDP vs TP 헷갈림 정리**
> - **데이터를 같이 받나**: FSDP는 칩마다 다른 배치(DP), TP는 칩마다 같은 배치(분업).
> - **계산 순간 W가 온전한가**: FSDP는 전체로 복원해 계산(결과도 전체), TP는 쪼갠 채 계산(결과도 반쪽).
> - "복원하면 메모리 같지 않나?" → 복원되는 건 **레이어 하나**지 모델 전체가 아니다. 대가로 레이어마다 통신 발생 = **메모리 ↔ 통신 교환.**

### 3.4 GSPMD

> **GSPMD = "이 텐서를 이 축으로 쪼개"라고 선언만 하면, 칩끼리의 통신을 XLA가 자동으로 채워주는 시스템.**

같은 행렬곱 `Y = X · W`, 칩 2개(C0, C1). **선언 한 줄만 바꾸면 DP도 TP도 된다.**

**선언 A: "X를 배치 축으로 분할, W는 복제" → DP/FSDP**
```
C0: X 위쪽 절반 + W 전체  → Y의 위쪽 절반을 독립 계산
C1: X 아래쪽 절반 + W 전체 → Y의 아래쪽 절반을 독립 계산
```
XLA 자동 추론: 각 칩이 독립 계산 가능 → **통신 0**. (W도 쪼개 저장했다면 계산 직전 all-gather를 컴파일러가 자동 삽입 = FSDP.)

**선언 B: "W를 열 축으로 분할, X는 복제" → TP**
```
C0: X 전체 + W 왼쪽 절반  → Y의 왼쪽 절반
C1: X 전체 + W 오른쪽 절반 → Y의 오른쪽 절반
```
XLA 자동 추론: 각 칩이 Y 반쪽만 생성 → 다음 레이어가 전체 Y를 원하면 **all-gather가 필요함을 스스로 알아채고 통신 삽입**.

**핵심 = sharding propagation (샤딩 전파).** 그래프 몇 군데에만 "이 텐서는 이렇게 쪼개져 있다"고 점 찍으면, XLA가 그 정보를 그래프 전체로 번지게 해 중간 텐서 샤딩을 추론하고, 안 맞는 경계마다 통신을 끼워넣는다. (스도쿠에서 숫자 몇 개만 채우면 나머지가 논리적으로 결정되는 것과 같다.)

| | Megatron (수동) | GSPMD (선언적) |
|---|---|---|
| 쪼개는 방식 | 사람이 코드로 명시 | 축 선언 한 줄 |
| 통신 삽입 | 사람이 직접 | XLA가 자동 추론·삽입 |
| DP↔TP 전환 | 코드 大수술 | 선언 한 줄 수정 |
| 3D 병렬화 | 셋을 손으로 짜맞춤 | mesh에 축 매핑하면 조합 자동 |

DP/TP/PP를 따로 짜는 게 아니라, **"어느 텐서를 mesh(칩 격자)의 어느 축에 매핑하느냐"라는 단일 언어**로 통합된다.

**대가**: 모양이 컴파일 타임에 고정돼야 하고, 샤딩 전파가 항상 최적은 아니라 사람이 힌트를 더 박아줘야 할 때가 있다.

## 4. ICI / OCS — 칩을 잇는 네트워크

이제 GSPMD가 끼워넣는 통신이 **실제로 무슨 선을 타고 흐르느냐**.

### 4.1 GPU 방식 (대비)

2계층 구조:
```
노드 안 (8장): NVLink 직접 연결 — 초고속
노드 사이:     NIC → 이더넷/InfiniBand 스위치 → NIC — 상대적으로 느림
```
- 노드를 넘으면 **스위치(라우터)를 거쳐** 패킷이 돈다 (RoCE/InfiniBand).
- NVLink 직결 도메인이 작다 (~8, NVL72면 72). 그 너머는 느린 이더넷. → TP를 노드 안에서만 쓰는 이유.

### 4.2 TPU: ICI (Inter-Chip Interconnect)

칩끼리 **스위치 없이 직접** 잇는다. 칩에 통신 링크가 박혀 옆 칩과 케이블로 직결.
```
GPU:  칩 → NIC → 스위치 → NIC → 칩   (라우터 경유)
TPU:  칩 ━━━━━━━━━ 칩              (직접, 라우터 없음)
```

칩을 **3D 격자(토러스)**로 엮는다 — 각 칩이 6방향(±x,±y,±z) 이웃과 직결, 가장자리는 반대편으로 감겨 도넛 모양.

**왜 격자/토러스?** GSPMD 통신 대부분이 이웃끼리 데이터를 차곡차곡 도는 패턴(all-reduce, all-gather). 임의의 먼 칩과 직통할 필요가 적으니, 모든 칩을 모든 칩에 잇는(비싼) 대신 이웃 직결 격자가 딱 맞다. 토러스면 어느 방향으로도 균등하게 돌릴 수 있다.

> NVLink가 ~72개에서 InfiniBand로 넘어가야 하는 반면, ICI는 그 약 100배 규모(최대 9,216칩)를 직접 연결한다. TP처럼 잦은 통신을 수천 칩 스케일에서도 견디는 이유.

### 4.3 OCS (Optical Circuit Switch, 광회로 스위치)

3D 격자를 케이블로 물리 고정하면 문제: 워크로드마다 필요한 칩 수가 다르고, 칩 고장 시 격자에 구멍이 나며, "격자 한 조각만 떼주기"가 안 된다.

> **OCS는 패킷을 라우팅하는 스위치가 아니다.** 빛을 거울로 물리적으로 반사해, 어느 칩 다발이 어느 칩 다발과 연결될지 **배선 자체를 바꾸는** 장치다.

비유: 보통 이더넷 스위치는 **교통경찰**(패킷마다 "넌 이쪽, 넌 저쪽" 매번 지시). OCS는 **철도 전철기(선로 변환기)** — 선로 자체를 미리 갈아끼워 놓고 그 위로 기차가 직진하게 한다. 한 번 연결을 맺으면 OCS는 빠지고 칩끼리 직통으로 흐른다.

**OCS가 가능하게 하는 것**:
1. **토폴로지 재단** — 9,216칩 ICI 망에서 4×4×4 큐브 단위로 더 작은 논리 슬라이스를 즉석 조립. 잡 크기에 맞춰 격자를 잘라줌.
2. **고장 우회** — 칩이 죽으면 배선을 바꿔 죽은 칩을 빼고 격자를 다시 꿰. (케이블 직결이면 불가능.)
3. **거리 극복** — 빛이라 구리선보다 멀리 가, 데이터센터 전체에 칩을 흩어놔도 격자로 묶임.

| | GPU (NVIDIA) | TPU (Google) |
|---|---|---|
| 칩 사이 | NIC → 스위치 → NIC | ICI 직결 (라우터 없음) |
| 토폴로지 | 트리형 (스위치 계층) | 3D 토러스 (이웃 직결) |
| 직결 도메인 | ~8\~72칩 | 최대 9,216칩 |
| 스위치 역할 | 매 패킷 라우팅 (교통경찰) | OCS = 배선 변경 (전철기) |
| 토폴로지 변경 | 고정 (물리 배선) | **소프트웨어로 재구성** |

ICI 직결 격자 덕에 "노드 안"의 범위가 수천 칩까지 넓어진다. GPU라면 느린 이더넷으로 떨어졌을 통신을 TPU는 빠른 직결로 유지 → 거대 모델을 수천 칩에 펼쳐도 GSPMD가 끼워넣는 통신이 병목이 안 됨.

## 5. Ironwood — 왜 "inference-first"인가

### 5.1 추론은 두 단계: Prefill vs Decode

"프랑스의 수도는" → "파리입니다" 생성 과정:

- **Prefill**: 입력 프롬프트 전체를 한 방에 처리해 첫 토큰 생성. 토큰이 여러 개 → 큰 행렬곱 → **시스톨릭 어레이가 꽉 참 → 계산 바운드.**
- **Decode**: 그 다음부턴 토큰 하나씩 생성. 한 번에 딱 한 토큰.

### 5.2 Decode가 어레이를 굶긴다

시스톨릭은 "크고 규칙적인 행렬곱일 때만 이득, 작으면 텅 빈다"(§1.6). Decode가 정확히 그 경우다.

```
Prefill: [토큰 100개] × [가중치]  → 256×256 어레이 꽉 참  ✓
Decode:  [토큰 1개]   × [가중치]  → 어레이 1줄만 쓰고 255줄이 놂  ✗
```

결정적 전환:
- **Prefill = 계산 바운드** (어레이가 일함)
- **Decode = 메모리 바운드** (어레이는 놀고, 토큰 하나 곱하려고 거대한 가중치 전체를 HBM에서 읽는 게 병목)

대전제(§0)대로 칩에서 비싼 건 메모리 읽기. Decode는 그 비싼 짓을 토큰마다 반복하는데 계산은 쥐꼬리. **시스톨릭의 최악 시나리오가 LLM 추론의 대부분(decode)이다.**

### 5.3 KV cache

토큰마다 앞 토큰 전체를 다시 계산하지 않으려고 이미 계산한 걸 저장·재사용 = **KV cache.** 긴 컨텍스트 + 많은 동시 사용자 = KV cache가 수백 GB.

→ 추론 하드웨어가 진짜 원하는 것은 **계산 괴물이 아니라**:
1. **거대한 HBM 용량** (KV cache + 가중치)
2. **거대한 HBM 대역폭** (decode가 가중치를 빨리 읽게)
계산력은 부차적.

### 5.4 Ironwood 스펙 = decode 메모리 바운드 대응

| Ironwood 선택 | 왜 (decode 관점) |
|---|---|
| 192GB HBM (Trillium의 6배) | KV cache + 가중치 담을 **용량** |
| 7.4 TB/s 대역폭 | decode가 가중치를 빨리 읽을 **대역폭** |
| FP8 native (TPU 최초) | 가중치를 작게 → 읽을 byte 감소 |
| 와트당 2배 효율 | 서빙은 24시간 → 전력이 곧 돈 |
| 9,216칩 pod, 42.5 ExaFLOPS | scale-out (ICI/OCS) |

이전 모든 TPU가 training 우선이었던 것과 달리, Ironwood는 처음으로 inference를 위해 바닥부터 설계됐다. 같은 시스톨릭 어레이라도 주변(메모리·네트워크)을 추론에 맞춰 재단한 것.

### 5.5 왜 경제학인가

> **학습은 한 번**(모델 하나를 비싸게 한 번 학습하면 끝).
> **추론은 영원**(수백만 사용자가 매일, 매 쿼리마다 돌림).

프론티어 모델 학습은 수천만 달러의 일회성 비용이지만, 매일 서빙하는 비용은 모델 수명 전체에 걸쳐 학습 예산을 훨씬 능가하는 지속 비용이다. AI가 연구 → 제품으로 넘어가며 **돈이 가장 많이 드는 곳이 학습 → 추론으로 이동**. 칩 설계 우선순위도 "학습을 빨리"에서 "**추론을 싸게**"로 이동했고, Ironwood는 그 이동에 베팅한 칩이다.

(맥락: Anthropic이 최대 100만 TPU 접근을 계획 — Claude를 매일 서빙하는 추론 비용이 거대하며, 이를 GPU보다 싸게 칠 수 있다면 막대한 절약. 대규모 추론에서 TPU·AWS 실리콘이 가격 대비 성능 우위를 점할 것이라는 분석이 이 맥락.)

## 6. 전체 종합

5개 세션은 사실 **하나의 명제**였다:

> **경직된 전용 하드웨어(시스톨릭) + 똑똑한 컴파일러(XLA·GSPMD) + 직결 네트워크(ICI/OCS)로, 추론을 GPU보다 싸게 친다.**

| 세션 | 무엇 | 한 줄 |
|---|---|---|
| 1. 시스톨릭 | 계산 효율 | 가중치 박고 데이터 흘려 메모리 왕복 제거 |
| 2. XLA | 컴파일 (칩 안) | 연산 융합으로 메모리 왕복 제거 (같은 강박, SW판) |
| 3. GSPMD | 샤딩 (칩 사이) | 축만 선언하면 통신을 컴파일러가 추론 |
| 4. ICI/OCS | 네트워크 | 스위치 없는 직결 격자로 통신을 싸게 |
| 5. Ironwood | 추론 경제학 | 위 전부를 decode 메모리 바운드에 맞춰 재단 |

관통 철학 둘: **"메모리 왕복을 없애라"** + **"경직된 하드웨어, 똑똑한 컴파일러."** NVIDIA의 "유연한 하드웨어 + 손으로 짠 커널 + 거대 생태계"와 정반대 베팅.

## 부록 B. 핵심 용어

- **MAC**: Multiply-Accumulate. 곱하고 누적하는 기본 연산.
- **weight-stationary**: 가중치를 PE에 고정하고 입력을 흘리는 데이터플로우.
- **staggering**: 행마다 입력 투입을 한 클럭씩 어긋내 타이밍 맞춤.
- **operator fusion**: 여러 연산을 한 커널로 합쳐 중간값 HBM 왕복 제거.
- **AOT (Ahead-Of-Time)**: 실행 전 미리 통째로 컴파일.
- **sharding**: 텐서를 여러 칩에 쪼개는 것.
- **sharding propagation**: 일부 샤딩 지정으로부터 나머지를 컴파일러가 추론.
- **all-gather / all-reduce**: 칩들이 조각을 모으거나(gather) 합산해 공유(reduce)하는 집합 통신.
- **ICI**: Inter-Chip Interconnect. TPU 칩 직결 링크.
- **OCS**: Optical Circuit Switch. 배선을 소프트웨어로 바꾸는 광스위치.
- **prefill / decode**: 추론의 입력 일괄 처리 단계 / 토큰 하나씩 생성 단계.
- **KV cache**: decode에서 앞 토큰의 key/value를 저장·재사용하는 캐시.
- **계산 바운드 / 메모리 바운드**: 병목이 연산력인지 / 메모리 대역폭인지.
